* Introduction

This document defines the conceptual architecture of the Axum Clean system
after introducing dynamic authorization and projection.

The system evolves from endpoint authorization toward:

#+begin_quote
Policy defines what reality an actor can see or modify.
#+end_quote

Authorization is implemented strictly inside the *Application Layer*.

---

* Architectural Evolution

** Previous Model

UseCase
→ load entity
→ check permissions
→ manually filter fields
→ build DTO

Problems:

- duplicated authorization
- fragile security
- DTO explosion
- developer responsibility for safety

---

** New Model

Policy → Scopes → Fields → Projection → Response

Authorization shapes data automatically.

---

* Conceptual Layers

| Layer       | Responsibility      |
| ----------- | ------------------- |
| Domain      | Business truth      |
| Application | Policy + projection |
| Interface   | Transport only      |

---

** Domain Layer

- Pure entities
- No roles
- No permissions
- No visibility logic

Domain answers:

#+begin_quote
What is true?
#+end_quote

---

** Application Layer

Contains:

- Policy Engine
- Scope Resolver
- Projection Builder
- Authorization Validation
- Reference Resolver

Answers:

#+begin_quote
Who may interact with what?
#+end_quote

---

** Interface Layer

- HTTP adapters
- serialization
- request parsing

No authorization decisions.

---

* Core Vocabulary

** Capability (Intent)

Describes action:

- READ
- UPDATE
- APPROVE
- MANAGE

---

** Scope (Security Meaning)

Semantic protection zones.

Examples:

- PublicProfile
- PrivateProfile
- Preferences
- SecurityCredentials
- SystemMetadata

Scopes explain *WHY* data is protected.

---

** Field (Data Precision)

Selectable unit of information.

Examples:

- profile.first_name
- preferences.language
- email
- version

Fields evolve frequently.

---

** Mapping (Bridge)

Every field belongs to exactly one scope.

Field → Scope

Example:

email → PrivateProfile
password → SecurityCredentials
version → SystemMetadata

This mapping is the single security truth.

---

* Unified Authorization Flow

1. Actor expresses intent
2. Policy evaluates context
3. Allowed scopes returned
4. Fields derived automatically
5. Projection shapes result

---

* Request → Response Lifecycle

1. HTTP request received
2. Controller invokes use case
3. Policy context created
4. Policy evaluated
5. Projection generated
6. Repository queried
7. Projection applied
8. Response returned

* Read Authorization

AllowedFields =
all fields whose scope ∈ AllowedScopes

Manual filtering disappears.

---

* Write Authorization (Unified Model)

UpdateFields ⊆ AllowedWritableFields

Unauthorized changes rejected BEFORE domain execution.

Example:

Author cannot approve own report.

Handled by policy constraints.

---

* Sensitive Fields Model

Sensitive data stops being special-case logic.

| Field    | Scope               |
| -------- | ------------------- |
| password | SecurityCredentials |
| version  | SystemMetadata      |
| email    | PrivateProfile      |

Security becomes structural.

* Dynamic Roles

Roles map dynamically to scopes:

Role → AllowedScopes

Changing DB configuration changes system behavior instantly.

No redeployment required.

---

* Advantages

- Structural security
- No accidental data leakage
- Dynamic authorization
- Cleaner use cases
- Multi-tenant ready
- GraphQL-like flexibility without GraphQL complexity
- Audit-friendly behavior

---

* Tradeoffs

- Higher abstraction complexity
- Requires mapping discipline
- Needs debugging visibility tools
- Slight runtime filtering cost

---

* Architectural Classification

This architecture aligns conceptually with:

- Capability-based access control
- Attribute-based access control
- Policy-driven systems
- Zanzibar-style reasoning models

---
