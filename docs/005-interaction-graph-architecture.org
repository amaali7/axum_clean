* Overview

This document defines the Interaction Graph model used by Axum Clean
to unify authorization, projection, and business interaction rules.

The system moves from permission checking toward:

#+begin_quote
Defining the set of interactions an actor may perform within the system.
#+end_quote

Authorization becomes a graph evaluation problem.

---

* Motivation

Traditional systems evaluate permissions per endpoint:

User → Role → Permission → Allow/Deny

Problems:

- rules scattered across use cases
- ownership rules duplicated
- difficult policy evolution
- hard to simulate behavior

The Interaction Graph solves this by modeling interactions explicitly.

---

* Core Idea

All system behavior is represented as:

Actor ── interacts via Intent ──> Resource

Each interaction becomes a node in a graph.

Policy evaluation determines whether an edge exists.

---

* Interaction Definition

Interaction =

| Component   | Meaning             |
| ----------- | ------------------- |
| Actor       | who performs action |
| Intent      | capability          |
| Resource    | target entity       |
| Context     | environment data    |
| Constraints | validation rules    |

---

Example:

Author --(Approve)--> Report

Constraint:

report.author_id ≠ actor.id

* Intents (Capabilities)

Intents represent business actions.

Examples:

- Read
- Update
- Create
- Delete
- Approve
- Review
- Manage

Intents are stable concepts independent of fields.

---

* Resources

Resources are domain aggregates:

- User
- Report
- Tenant
- Role
- Permission

The graph never exposes database structure.

---

* Constraints

Constraints express dynamic rules.

Examples:

- ownership rules
- tenant isolation
- temporal restrictions
- workflow state validation

---

* Interaction Evaluation Flow

1. Actor requests intent
2. Interaction node located
3. Policy resolves scopes
4. Constraints evaluated
5. Decision produced
6. Projection derived

---

* Interaction Evaluation Sequence

#+begin_src plantuml

InteractionEngine -> PolicyEngine

InteractionEngine -> ConstraintEvaluator
ConstraintEvaluator --> InteractionEngine

InteractionEngine -> ProjectionBuilder
ProjectionBuilder --> UseCase

UseCase -> Controller : response

* Relationship With Policy Engine

Policy answers:

Which interactions exist?

Interaction Graph answers:

Is this specific interaction valid now?

Policy = structure
Graph = execution

* Read vs Write Unification

Reads and writes follow identical logic.

Read:

Actor --(Read)--> Resource

Write:

Actor --(Update)--> Resource

Both evaluated through same interaction graph.

* Governance Lifecycle

Policy lifecycle:

Draft
→ Simulation
→ Review
→ Approval
→ Active
→ Archived

Interaction graph automatically updates after activation.

---

* Advantages

- Unified authorization model
- Eliminates special-case logic
- Supports dynamic policies
- Enables safe experimentation
- Simplifies use cases
- Improves auditability

---

* Tradeoffs

- Higher conceptual abstraction
- Requires visualization tools
- Needs policy debugging interface

---

* Mental Model

System behavior becomes:

#+begin_quote
A graph of allowed interactions instead of permission checks.
#+end_quote

Actor perceives only reachable nodes.

---

* Relationship to Overall Architecture

Policy Engine → defines edges
Interaction Graph → validates edges
Projection System → shapes data
Domain → executes business logic

---

* Architectural Outcome

The system becomes:

Policy-Governed Interaction Platform

Authorization defines possible actions.
Projection defines visible data.
Domain enforces business invariants.
